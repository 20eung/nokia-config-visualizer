# Name Dictionary - Plan Document

> **Feature**: Name Dictionary (이름 사전)
> **Version**: v4.1.0
> **Status**: ✅ Implemented
> **Type**: Retrospective PDCA Documentation

---

## 1. Feature Overview

### 1.1 Summary

Name Dictionary는 Nokia Config의 description 필드에서 AI가 자동으로 엔티티(고객명, 지역, 서비스명, 장비명 등)를 추출하여 이름 사전을 구축하는 기능입니다. 사용자는 원본 영문 토큰뿐만 아니라 한국어 이름이나 별칭으로도 서비스를 검색할 수 있습니다.

### 1.2 Key Capabilities

1. **AI 자동 생성**: AWS Bedrock Claude를 사용하여 description에서 의미 있는 엔티티 추출
2. **이름 변형 생성**: 각 엔티티에 대해 shortName, longName, koreanName, aliases 자동 생성
3. **증분 구축**: 기존 사전 항목을 보존하면서 새로운 토큰만 추가
4. **AI 챗봇 연동**: 한국어/별칭으로 질문해도 원본 토큰과 매칭하여 서비스 검색
5. **전역 단일 사전**: 모든 config 조합에서 동일한 사전 공유

### 1.3 Example Use Case

```
사전 등록: 11st → 한국어: 11번가, 별칭: 일일번가
AI 검색:
  - "11st 서비스 보여줘" ✅
  - "11번가 서비스 보여줘" ✅
  - "일일번가 서비스 보여줘" ✅
```

## 2. Business Value

### 2.1 Problem Statement

**현재 상황**:
- Config description은 영문 약어 중심 (예: `Cust-A_HQ_1G`, `Nokia-PE1_to_CE1`)
- 한국 사용자는 영문 약어를 외우거나 매번 찾아봐야 함
- 동일 엔티티를 다양한 이름으로 표현 (11st, 11번가, 일일번가)
- AI 챗봇이 한국어 질문을 원본 토큰과 매칭하기 어려움

**문제 영향**:
- 서비스 검색 시 정확한 영문 토큰을 알아야만 검색 가능
- 한국어 자연어 검색이 불가능하여 사용성 저하
- 새로운 사용자의 학습 곡선 증가

### 2.2 Business Benefits

1. **사용성 향상**: 한국어/별칭으로 자연스럽게 검색 가능
2. **학습 곡선 감소**: 영문 약어를 외울 필요 없음
3. **AI 챗봇 정확도 향상**: 다양한 표현을 원본 토큰과 매칭
4. **생산성 증가**: 빠른 서비스 검색으로 작업 시간 단축
5. **확장성**: 프로젝트가 커져도 사전을 누적 구축

### 2.3 Target Users

- **Primary**: Nokia Config 분석 담당자 (한국어 사용자)
- **Secondary**: AI 챗봇을 사용하는 모든 사용자

### 6.2 Data Flow

#### 6.2.1 AI 자동 생성 Flow

```
1. User uploads Config files
2. descriptionExtractor.ts extracts unique descriptions
3. User clicks "AI 자동 생성" button
4. DictionaryEditor.tsx → dictionaryApi.generateDictionary()
5. POST /api/dictionary/generate (descriptions)
6. dictionaryGenerator.ts → AWS Bedrock (Claude Sonnet 4)
7. AI extracts entities and generates name variants
8. Response: { entries: DictionaryEntry[] }
9. DictionaryEditor.tsx filters duplicates and adds new entries
10. User clicks "저장"
11. PUT /api/dictionary (full dictionary)
12. dictionaryStore.saveDictionary() → /app/data/dictionary.json
```

#### 6.2.2 사전 로드 Flow

```
1. ServiceListV3.tsx mounts
2. useEffect → dictionaryApi.loadDictionaryFromServer()
3. GET /api/dictionary
4. dictionaryStore.loadDictionary() → read /app/data/dictionary.json
5. Response: NameDictionary
6. ServiceListV3.tsx stores in state
7. AI 챗봇 활성화 시 DictionaryCompact 형식으로 전달
```

#### 6.2.3 한국어 검색 Flow

```
1. User activates AI chatbot
2. ServiceListV3.tsx converts NameDictionary → DictionaryCompact
3. User types Korean query: "11번가 서비스 보여줘"
4. AIChatPanel.tsx → POST /api/chat (query + DictionaryCompact)
5. systemPrompt.ts instructs Claude to use dictionary
6. Claude matches "11번가" → "11st" (originalToken)
7. Response: { selectedKeys: ["nokia-1:service:11st"] }
8. ServiceListV3.tsx selects matching services
9. V3Page.tsx generates diagram
```

### 7.3 Key Technical Decisions

#### Decision 1: 전역 단일 사전 vs Fingerprint 기반 사전

**선택**: 전역 단일 사전
**이유**:
- Config 파일 조합이 자주 바뀌는 환경
- 사전을 한 번 구축하면 모든 config에서 재사용
- Fingerprint 방식은 조합이 바뀔 때마다 사전 다시 구축 필요

**Trade-offs**:
- ✅ 장점: 사전 재사용, 누적 구축 가능
- ⚠️ 단점: 여러 프로젝트 동시 사용 시 혼재 가능 (현재는 단일 프로젝트 환경)

#### Decision 2: AWS Bedrock Claude vs Other AI

**선택**: AWS Bedrock Claude Sonnet 4
**이유**:
- 기존 AI 챗봇에서 사용 중 (v3.3.0)
- 한국어 처리 능력 우수
- 긴 컨텍스트 지원 (8192 tokens)
- AWS 인프라와 통합 용이

**Trade-offs**:
- ✅ 장점: 일관된 AI 경험, 한국어 지원
- ⚠️ 단점: AWS 의존성, 비용

#### Decision 3: Server File Storage vs Database

**선택**: Server File Storage (`/app/data/dictionary.json`)
**이유**:
- 단순한 JSON 구조
- 읽기/쓰기 빈도 낮음
- Database 오버헤드 불필요
- Docker volume으로 충분한 지속성

**Trade-offs**:
- ✅ 장점: 단순성, 배포 용이
- ⚠️ 단점: 동시성 제어 없음 (현재는 단일 사용자 환경)

- ✅ Docker volume으로 데이터 보호
